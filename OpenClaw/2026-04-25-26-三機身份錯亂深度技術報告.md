# 2026-04-25~26 三機身份錯亂根本原因完整技術報告

> 作者：Claude Code（克勞德助教）  
> 日期：2026-04-26  
> 補充說明：本篇是 Grok 版「2026-04-26-Tri-Agent-Identity-Confusion.md」的深度技術版，記錄從程式碼層找到的真正根本原因。

---

## 一、問題症狀

1. 教練叫「學弟」，卻是「學妹」回應
2. 三機在群組都自稱「學長」或身份混淆
3. 修改 SOUL.md 無效——改了也沒用
4. Kong 的 cron relay 不斷 timeout（30 秒後錯誤）
5. 1號機（學長）認為 2/3號機在 VPS，而非 Mac mini

---

## 二、根本原因（從程式碼層找到）

### 問題 1：hook handler.js 的 BOT_ID 標籤對調

**檔案位置：**
- `/Users/bymyway/.openclaw-kong/hooks/bot-relay/handler.js`
- `/Users/bymyway/.openclaw-peipei/hooks/bot-relay/handler.js`

**錯誤內容：**
```javascript
// kong 的 handler.js 寫的是：
const BOT_ID = '3号機-孔大哥';  // ❌ 應該是 2號機

// peipei 的 handler.js 寫的是：
const BOT_ID = '2号機-佩佩';    // ❌ 應該是 3號機
```

這導致 BOT_MESSAGES.md 的記錄標籤全部錯誤，影響所有夥伴感知。

---

### 問題 2：bot-relay-inbound hook 注入錯誤身份（最關鍵）

**這就是為什麼修改 SOUL.md 完全無效的原因。**

`bot-relay-inbound/handler.js` 在每一條訊息進來之前，會把身份識別文字塞進 context。這段注入的優先級高於 SOUL.md。

**Kong 的 inbound handler 錯誤內容：**
```javascript
const BOT_ID = '3号機-孔大哥';   // ❌
// 注入文字寫的是：
'你是 **小龍蝦 3號機（學妹）**，Bot: @coachwu_lenovo_bot'  // ❌ 完全是學妹的資料！
```

**修正後：**
```javascript
const BOT_ID = '2号機-孔大哥';
'你是 **小龍蝦 2號機（學弟）**，Bot: @CoachWu_openclaw_bot，服務孔大哥（峰哥）。'
```

> ⚠️ **教訓：** 以後若 SOUL.md 改了沒用，第一時間檢查 `bot-relay-inbound/handler.js`，它每次都會覆蓋你的設定。

---

### 問題 3：sessions_send 跨進程路由失敗

**三機是獨立進程（ports 18789/18790/18793），sessions_send 只在同一進程內有效。**

1號機的 SOUL.md 原本寫著要用 `sessions_send` 呼叫學弟，
但學妹（peipei）恰好在大群組有 active session，
所以訊息被路由到學妹，出現「叫學弟，學妹回應」的現象。

**修正：** 跨機通訊一律改用 `BOT_RELAY.json`：
```json
{
  "messages": [{
    "from": "學長",
    "bot": "2",        // "2" = 學弟, "3" = 學妹
    "content": "任務內容",
    "status": "pending"
  }]
}
```

---

### 問題 4：Cron relay 401 Unauthorized → timeout

Kong 的 cron AI 收到任務後，**誤以為要自己呼叫 Telegram API**，
用 curl 發送，但沒有正確的 Bot Token，所以一直 401，最後 timeout。

**修正：** cron payload 加入明確禁止：
```
直接輸出文字：「學弟（2號機）已收到...」
⚠️ 禁止手動呼叫 Telegram API 或 curl。系統會自動把你的輸出發到群組。
```

---

### 問題 5：1號機誤判 2/3號機在 VPS

1號機做了 filesystem 驗證：`ls ~/.openclaw-kong/` → 找不到 → 結論「可能在 VPS」。

**事實：** 三台全在同一台 Mac mini 4，只是目錄在：
- 1號機：`/Users/bymyway/.openclaw/`
- 2號機：`/Users/bymyway/.openclaw-kong/`
- 3號機：`/Users/bymyway/.openclaw-peipei/`

1號機從自己的進程看不到另外兩個目錄（權限或工作目錄問題），但不代表不存在。

**修正：** SOUL.md 加入明確路徑 + 禁止用 filesystem 驗證的規則。

---

### 問題 6：Session 快取導致修改不生效

修改 SOUL.md 後必須清除 session 快取並重啟，否則 bot 仍讀舊內容：

```bash
# 清除快取（三機都要做）
rm -f ~/.openclaw/agents/main/sessions/*.jsonl
echo '{}' > ~/.openclaw/agents/main/sessions.json

# 重啟
launchctl kickstart -k system/com.openclaw.bot.main
# 或用 gateway 管理
```

## 四、其他收穫

### OpenCode 桌面版新增 MiniMax M2.7

設定路徑：`~/.config/opencode/opencode.jsonc`

```jsonc
"minimax_official": {
  "npm": "@ai-sdk/openai-compatible",
  "options": {
    "baseURL": "https://api.minimax.io/v1",
    "apiKey": "sk-cp-0_..."
  },
  "models": { "MiniMax-M2.7": {} }
}
```

同時從 `disabled_providers` 移除 `"minimax"`。

### NVIDIA Nemotron Super 49B 無法在 OpenCode 使用

**原因：** OpenCode 每次請求都附帶 `tools` 陣列（function calling）。
Nemotron Super 49B 收到 tools 時回傳 `content: null, tool_calls: []`，
OpenCode 解讀為「無回應」，介面一片空白。

**結論：** 這是模型相容性問題，無法從使用者端修復。建議移入 `blacklist`。

## 五、最重要的三條教訓

1. **SOUL.md 改了沒用？先查 `bot-relay-inbound/handler.js`**——它在每條訊息前覆蓋身份。
2. **跨機通訊禁用 sessions_send**——只用 BOT_RELAY.json。
3. **改完設定必須清 session 快取再重啟**——否則白改。
